Proyecto frontend relacionado: mercapleno-react


1) VISTA GENERAL DE ARQUITECTURA
--------------------------------
Este backend esta hecho en NestJS con arquitectura modular.

Patron base por modulo:
- controller: recibe HTTP
- service: implementa logica de negocio
- dto: valida payload de entrada
- module: registra controller/service en Nest

Capas de datos:
- Prisma ORM: usado en auth, users-admin, products
- mysql2 pool: usado en inventory, sales, reports (consultas SQL y transacciones)

Seguridad:
- JWT guard global (todas las rutas protegidas por defecto)
- rutas publicas solo con decorador @Public()

3.1) CORE
----------
- src/main.ts
  Funcion:
  - bootstrap de la aplicacion Nest
  Bloques de codigo:
  1. Crear app:
     const app = await NestFactory.create(AppModule)
  2. Prefijo global API:
     app.setGlobalPrefix('api')
     Resultado: todas las rutas quedan en /api/...
  3. CORS:
     app.enableCors()
  4. ValidationPipe global:
     - whitelist: elimina campos no permitidos
     - forbidNonWhitelisted: rechaza campos extras
     - transform: convierte tipos automatico
  5. Swagger:
     - genera docs en /api/docs
  6. Listen:
     - usa envs.port

- src/app.module.ts
  Funcion:
  - modulo raiz que conecta todos los modulos
  Bloques:
  1. imports: MysqlModule, PrismaModule, AuthModule, etc.
  2. providers globales:

3.5) AUTH
----------
- src/auth/auth.module.ts
  Funcion:
  - registra JWT/Passport + controller/service auth

- src/auth/auth.controller.ts
  Funcion:
  - define endpoints HTTP de auth
  Bloques (rutas):
  - POST /api/auth/register
  - POST /api/auth/login
  - POST /api/auth/verify-email
  - POST /api/auth/resend-verification
  - POST /api/auth/request-password-reset
  - POST /api/auth/reset-password
  - POST /api/auth/logout

- src/auth/auth.service.ts
  Funcion:
  - logica auth completa
  Bloques:
  1. helpers privados:
     - generateCode() (codigo 6 digitos)
     - hashCode() SHA256
     - buildExpiresAt()
     - isExpired()
  2. register():
     - valida duplicados email/doc
     - hash password bcrypt
     - crea usuario
     - crea codigo verificacion hash
     - intenta enviar email
  3. login():
     - busca usuario
     - compara password
     - exige email_verified
     - genera JWT
  4. verifyEmail():
     - valida codigo y expiracion
     - marca correo verificado
  5. resendVerification():
     - regenera codigo y envia correo
  6. requestPasswordReset():
     - genera codigo de recuperacion
  7. resetPassword():
     - valida codigo + expiracion
     - guarda nueva password hash
  8. logout():
     - respuesta de cierre lado cliente

- src/auth/guards/jwt-auth.guard.ts
  Funcion:
  - guard global JWT
  Bloque:
  - si ruta tiene @Public() => permite sin token
  - si no => exige Bearer JWT

- src/auth/strategies/jwt.strategy.ts
  Funcion:
  - valida payload JWT y crea user auth del request

- src/auth/decorators/public.decorator.ts
  Funcion:
  - marca endpoint como publico

- src/auth/decorators/current-user.decorator.ts
  Funcion:
  - extrae req.user en controllers

- src/auth/interfaces/auth-user.interface.ts
  Funcion:
  - tipo del usuario autenticado (id, id_rol, email)

5) APIS - COMO QUEDA FUNCIONANDO DESPUES DE LOS ARREGLOS
---------------------------------------------------------
Base URL backend:
- http://localhost:4000/api

Swagger:
- http://localhost:4000/api/docs

Auth:
- JWT guard global activo
- rutas publicas: auth/* (register/login/verificacion/reset)
- rutas protegidas: admin/users, productos, movimientos, sales, reports

Arreglos aplicados a APIs:
1. Admin users update:
   - antes: frontend usaba PUT y backend solo PATCH
   - ahora: backend acepta PATCH y PUT

2. Sales orders metodo de pago:
   - antes: DTO esperaba int y frontend mandaba M1/M2
   - ahora: DTO acepta string y service normaliza/valida metodo

3. Prisma startup:
   - antes: si DB no estaba arriba, backend se caia en onModuleInit
   - ahora: backend arranca y loggea error; endpoints que dependen DB fallan hasta que DB suba

4. Frontend integrado con backend:
   - URLs API centralizadas en frontend (apiConfig)
   - sin hardcode disperso de localhost:4000
   - usuarios admin usa PATCH
   - inventario usa servicio dedicado correcto
   - productos admin manda Authorization token


6) FLUJO FINAL (END-TO-END)
----------------------------
1. Usuario se registra
   - POST /api/auth/register
   - backend crea usuario + codigo email

2. Usuario verifica correo
   - POST /api/auth/verify-email

3. Usuario hace login
   - POST /api/auth/login
   - recibe token JWT

4. Frontend guarda token y lo envia como Bearer en rutas protegidas

5. Modulos operativos:
   - admin/users CRUD
   - productos CRUD
   - movimientos entradas/salidas
   - ventas y decremento stock
   - reportes JSON + PDF
